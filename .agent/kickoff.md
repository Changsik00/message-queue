# Agent Kickoff Instructions

> **중요**: AI Agent는 새로운 대화나 작업을 시작할 때 항상 이 문서를 가장 먼저 상기하고, 문맥(Context)을 완벽히 복구한 후 행동해야 합니다. 이전에 대화가 끊겼다가 다시 켜더라도 방황하지 않기 위한 지침입니다.

## 1. 문맥 복구 및 정찰 (Context Recovery)
- 가장 먼저 사용자가 지시한 `spec-XXX` 디렉토리(`/specs/spec-XXX/`)로 이동하여 `spec.md`, `plan.md`, `task.md`, `walkthrough.md`를 **전부 꼼꼼히 읽어서** 현재 상태와 잔여 작업을 파악하세요.
- 이전 작업 기록을 바탕으로 내가 지금 문서화 단계인지 코딩 단계인지 정확히 파악해야 합니다.

## 2. 작업의 두 단계 분리 (절대 원칙)
사용자의 허락 없이 멋대로 코드를 작성하거나 수정하지 마세요. 작업은 완전히 분리된 두 단계로 나뉩니다.

### Phase A: 문서화 단계 (Documentation)
1. 요구사항을 들으면 `/specs/spec-XXX/` 하위에 템플릿(`.agent/templates/`)을 복사합니다.
2. `spec.md`, `plan.md`, `task.md` 문서를 작성합니다.
3. 문서 작성이 완료되면 **반드시 작업을 멈추고 사용자에게 허락(Approval)을 구하세요.**
4. 사용자가 문서에 동의하기 전까지 절대 `git checkout -b` 로 브랜치를 파거나 소스코드를 건드리지 마세요.

### Phase B: 코딩 단계 (Autonomous Coding)
1. 사용자의 "허락"이 떨어지면, 그때 브랜치를 생성(`git checkout -b spec-XXX/short-desc`)하고 코딩을 시작합니다.
2. 이때부터는 **사용자에게 일일이 묻지 말고 `task.md`의 체크박스 순서대로 멈춤 없이(Autonomous) 쭉 코딩을 진행**하세요.
3. 체크박스 1개가 끝날 때마다 자동으로 Git Commit을 수행하세요. 
  - **주의 사항**: `git add .` 등의 명령어를 사용할 때는 반드시 사전에 최상위 디렉토리에 `.gitignore` 파일이 존재하는지 확인하고, `node_modules`, `venv`, `__pycache__`와 같은 대용량 의존성/캐시 폴더가 절대로 저장소에 커밋되지 않도록 통제해야 합니다.
4. 구현이 완료되면 반드시 **Agent가 먼저 자체적으로 로컬 테스트 및 동작 검증**을 수행하고 확인해야 합니다.
5. 테스트까지 끝나면 해당 스펙 디렉토리 안에 `pr_description.md` 파일을 작성합니다.
6. 작업 내역을 리모트 브랜치로 푸시한 뒤, 로컬 환경에 설치된 `gh` (GitHub CLI) 명령어를 사용하여 **직접 PR을 생성**(`gh pr create --title ... --body-file ...`)하세요.
7. 모든 체크박스가 끝나고 실제 GitHub PR 생성까지 완전히 끝나면 그때 사용자에게 완료(PR 링크 포함)를 보고하세요.
