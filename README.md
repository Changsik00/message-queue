# 🧪 Message Queue Lab — 시나리오 기반 MQ 심층 비교

> **"같은 시나리오, 다른 MQ"** — 단편적인 튜토리얼을 넘어, **하나의 현실적인 비즈니스 시나리오**를 Kafka, RabbitMQ, BullMQ, MQTT로 각각 구현하며 기술적 결정을 내리는 안목을 기르는 실험 프로젝트입니다.

---

## 🎯 왜 이 프로젝트를 하는가? (학습 목표)

메시지 큐(MQ)를 처음 배울 때 가장 큰 함정은 "각 기술을 따로 떼어내어 헬로월드만 찍어보고 끝난다"는 것입니다.
이 프로젝트는 철저히 **"현업에서 내가 아키텍처를 결정해야 할 때, 무엇을 기준으로 선택할 것인가?"** 에 답을 내리기 위해 설계되었습니다.

1. **시나리오 기반 체감**: 결제, 재고, 알림이 얽힌 '이커머스 파이프라인'이라는 동일한 조건 속에서 각 MQ를 교체해 봅니다.
2. **한계 테스트**: 단순히 돌아가는 것을 넘어, 서버를 강제로 죽여보고(장애 복구), 데이터를 왕창 쏟아부어보며(부하 테스트) 스펙의 한계를 직접 측정합니다.
3. **선택의 근거 마련**: "우리 회사 결제 시스템에 왜 Kafka 대신 RabbitMQ를 써야 하는가?"를 남에게 데이터와 시나리오로 설득할 수 있는 논리를 씁니다.

---

## 🛒 핵심 비교 시나리오: "이커머스 주문 처리 파이프라인"

가장 보편적이면서도, MQ의 특성(유실 방지, 재처리, 대용량 통신 등)이 극명하게 드러나는 시나리오입니다.

```text
[API 서버] (주문생성 POST)
    │
    ▼ publish(OrderEvent)
[Message Queue] ───────────▶ (목적에 따라 Kafka, RabbitMQ, BullMQ, MQTT 교체)
    │
    ├──▶ [결제 Worker]    → "반드시 1번만 결제되어야 함" (데이터 유실 절대 불가)
    ├──▶ [재고 Worker]    → "재고 차감이 실패하면?"
    └──▶ [알림 Worker]    → "알림은 실패 시 몇 번 더 시도해도 됨" (단순 Job 처리)
```

이 구조에서 **API와 Worker의 비즈니스 로직은 고정**한 채, 중앙의 **MQ만 교체**하면서 동작과 성능의 차이를 비교합니다.

---

## 🔬 무엇을 던지고, 무엇을 알아낼 것인가?

단순 구현을 넘어 아래의 6가지 핵심 질문에 대한 해답을 데이터로 도출합니다.

1. **처리량 (Throughput)**: 초당 1만 건의 주문 폭주 시나리오에서 어떤 MQ가 가장 잘 버티는가?
2. **지연시간 (Latency)**: 버튼을 누르고 결제가 완료되기까지 걸리는 시간이 가장 짧은 MQ는 무엇인가? (P95, P99 지연율 측정)
3. **유실 방어 (Fault Tolerance)**: 처리 도중 컨테이너의 전원을 뽑아버리면 내 주문 데이터는 사라지는가? (Ack/Nack 메커니즘 검증)
4. **결함 극복 (DLQ)**: 에러가 발생한 주문들만 따로 모아두고 수동으로 재처리할 수 있는가?
5. **과거 회귀 (Replay)**: "어제 발생한 주문 1,000건을 다시 결제 승인 로직에 태워라"라는 요구사항을 해결할 수 있는 MQ는 무엇인가?
6. **순서 보장 (Ordering)**: 100건의 주문이 생성된 순서와 100% 동일하게 소비자에게 도착하는가?

## 🗺️ 프로젝트 진행 로드맵 (Phase 1 ~ 4)

각 단계별 상세한 목표와 Task는 `backlog/` 디렉토리에 정의되어 있습니다.

| Phase | 단계 | 목표 | 기대 효과 | 링크 |
|---|---|---|---|---|
| **Phase 1** | **환경 구축** | Docker 인프라 구성 및 코어 아키텍처 설계 | **비교 기준점 확립** (동일 DB, 동일 이벤트 규격 마련) | [phase1.md](./backlog/phase1.md) |
| **Phase 2** | **교체 구현** | 4종 MQ x 2언어(Python/Node) 구현 완료 | **각 MQ의 설계 철학 이해** (Queue vs Topic vs Job) | [phase2.md](./backlog/phase2.md) |
| **Phase 3** | **극한 실험** | k6 부하 테스트 및 장애 모의 실험 | **정량적 데이터 확보** (초당 처리량, 지연 차트 생성) | [phase3.md](./backlog/phase3.md) |
| **Phase 4** | **전략 도출** | 실무 시나리오 기반의 아키텍처 선택 가이드 작성 | **현업 적용 능력 확보** (Saga, Webhook 등 매칭) | [phase4.md](./backlog/phase4.md) |

```
/message-queue

│
├── docs/                        # 📚 학습 및 결과 문서

│   └── strategy/                #   [Phase 4] 현업 실무 사례 기반 MQ 선택 가이드 (의사결정 트리)
│
├── api-server/                  # 🚀 주문 생성 API (Python FastAPI / Node.js Express)
├── workers/                     # 👷 이벤트 처리 Worker (언어별, MQ별 분리 구현)
├── mq/                          # ⚙️ 각 MQ별 로컬 설정 및 구성 스크립트
│
├── load-test/                   # 🔥 k6 / Locust 부하 테스트 스크립트
└── results/                     # 📊 실험 결과 시각화 차트 및 CSV
```
