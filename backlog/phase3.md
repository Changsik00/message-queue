# Phase 3 — 실험 & 측정

> **기간**: Week 4 | **상태**: 🔲 대기

---

## 🎯 이 Phase의 목적

Phase 2에서 "MQ가 돌아가게 만들었다"면,
Phase 3 에서는 **"돌아가는 MQ들이 실제로 얼마나 다른지"** 데이터로 증명한다.

코드가 작동한다고 해서 "좋은 MQ"를 선택한 게 아니다.
현업에서는 **"이 MQ로 우리 트래픽을 감당할 수 있는가?"**, **"장애 시 데이터를 잃지 않는가?"** 가 핵심이다.

이 Phase의 결과물(CSV, 측정 데이터)은 Phase 4에서 "MQ 선택 기준"을 정립하는 근거 데이터가 된다.

---

## 🧠 배경 지식 — 왜 이 항목들을 측정하는가?

### 처리량(Throughput)을 측정하는 이유
> "우리 서비스에 초당 1만 건 주문이 들어오기 시작했다. 현재 MQ가 버틸 수 있는가?"
처리량은 MQ 선택의 첫 번째 필터다.

### 지연시간(Latency)을 측정하는 이유
> "주문 버튼을 눌렀는데 결제 결과가 5초 후에 나온다. 이유가 뭔가?"
**실시간 결제 피드백**이 필요한 서비스라면 P95, P99 Latency가 매우 중요하다.

### 장애 복구(Fault Tolerance)를 테스트하는 이유
현업에서 장애 시 **데이터가 유실되는가, 중복 처리되는가**의 차이를 가른다.

### 재처리(Replay)를 테스트하는 이유
> "어제 로직 버그로 데이터를 잘못 썼다. 이벤트를 롤백해 다시 처리할 수 있는가?"
Kafka 고유의 기능과 나머지 MQ 간의 극명한 한계를 확인한다.

## ✅ Spec 목록 (MVP 분리)

아래 3개의 Spec은 각 성능 및 신뢰성 테스트를 작고 독립적인 단위로 진행하며, 측정 및 분석이 끝날 때마다 PR 단위로 머지합니다.

### 🚀 Spec 3-001. 부하 및 지연시간 성능 테스트 (Performance MVP)
> **동작 목표**: k6 기반 부하 테스트를 통해 4종의 성능 지표(TPS, Latency 분포)를 데이터(CSV)로 추출합니다.
- [ ] **구현**: `load-test/order-test.js` (k6 자바스크립트 스크립트 작성) 및 각 MQ를 도는 테스트 자동화.
- [ ] **측정 (Throughput)**: 100 → 1,000 → 5,000 TPS 단계별 부하시 처리 한계점 확인.
- [ ] **측정 (Latency)**: API에 요청이 도달한 시간과 워커가 처리한 시간(`consumed_at - publishedAt`) 바탕으로 중앙값(P50), P95, P99 분석.
- [ ] **검증**: `results/throughput.csv`, `results/latency.csv` 파일에 MQ별 8개 지표 적재 여부 확인.

### 💥 Spec 3-002. 장애 및 순서 보장 스트레스 테스트 (Fault Tolerance MVP)
> **동작 목표**: 처리 워커를 강제 종료(kill)하거나 MQ 브로커 재기동 시 데이터 유실/중복/재처리 기능이 작동하는지 직접 시연합니다.
- [ ] **강제 종료 테스트**: 워커 프로세스 강제 `kill -9` 후 재구동 시, Offset, DLQ 등에 의한 재처리 대응 방식 확인.
- [ ] **Replay 리플레이 테스트**: Kafka Consumer Group 의 offset 강제 리셋을 통한 1시간치 데이터 재처리 검증. (타 MQ의 불가능성 교차 검증)
- [ ] **순서 보장 테스트**: Kafka 파티션, RabbitMQ 단일 워커 구성 등에서 병렬 처리 중 순서 역전 현상이 일어나는지 검증.
- [ ] **검증**: `results/fault-tolerance.md`, `results/ordering.md` 에 결과 레포트 명세.

### 📊 Spec 3-003. 데이터 시각화 및 리포팅 (Docs)
> **동작 목표**: Python pandas나 matplotlib(또는 비슷한 도구)를 이용해 Phase 4의 기초가 될 핵심 차트를 산출합니다.
- [ ] **구현**: 수집된 `.csv` 스탯을 읽어 P95 Latency 차트 형태(`latency-p95-chart.png` 등)로 산출.
- [ ] **검증**: README나 리포트를 본 누구나 한 눈에 TPS와 레이턴시 승자 및 패자(?)를 구별할 수 있어야 함.
